# 开发计划：`parse_iso8601` — Julian helper 反向解析闭合

> 起稿：2026-04-21（第十四轮）  
> 前置：`docs/plans/2026-04-21-julian-date-helper-plan.md` 已实现 `format_iso8601` 输出方向，并在"未纳入本轮"中明确"反向解析（parse_iso8601）等到 UI 真有用户输入时间字符串场景再加"。本轮把这条 helper 反向链路补上，让 `format → parse → format` round-trip 闭合。

## 动机

`format_iso8601(dt)` 已经能从 `DateTimeUtc` 产出 `"YYYY-MM-DDTHH:MM:SSZ"` 字符串。但相反方向缺失：UI 层若提供"用户输入 timestamp"功能 / 命令行参数解析 / 配置文件读取 timestamp，需要从字符串回到 `DateTimeUtc`。

无依赖（不引入 chrono）的实现：手写严格的 `YYYY-MM-DDTHH:MM:SSZ` 格式解析（按字符 / 分段 split / 整数 parse）。约 30 行。

## 目标

1. `crates/h7cad-native-model/src/julian.rs` 新增：
   - `pub fn parse_iso8601(s: &str) -> Option<DateTimeUtc>`
2. 严格格式：`YYYY-MM-DDTHH:MM:SSZ`
   - 19 字符固定长度
   - 分隔符必须精确：`-` / `T` / `:` / `Z`
   - 字段范围基础校验（month 1-12 / day 1-31 / hour 0-23 / minute 0-59 / second 0-60 含闰秒位置但不真实模型化）
3. `lib.rs` 重新导出
4. 单测：valid / invalid / round-trip 共 4-5 条

## 非目标

- 不接受 fractional seconds（`.123` 后缀）
- 不接受 timezone offset（`+08:00` / `-05:00`）—— 只 UTC `Z`
- 不接受 lower-case `z` / `t`（严格 upper case）
- 不做闰秒物理验证（second = 60 接受但语义不变）
- 不接受短格式（`YYYY-MM-DD`）
- 不接受 BC 日期（year 必须 0001-9999）

## 关键设计

### 解析

```rust
pub fn parse_iso8601(s: &str) -> Option<DateTimeUtc> {
    if s.len() != 20 {
        return None;
    }
    let bytes = s.as_bytes();
    if bytes[4] != b'-'
        || bytes[7] != b'-'
        || bytes[10] != b'T'
        || bytes[13] != b':'
        || bytes[16] != b':'
        || bytes[19] != b'Z'
    {
        return None;
    }
    let parse_u = |from: usize, to: usize| -> Option<u32> {
        s[from..to].parse::<u32>().ok()
    };
    let year = s[0..4].parse::<i32>().ok()?;
    let month = parse_u(5, 7)?;
    let day = parse_u(8, 10)?;
    let hour = parse_u(11, 13)?;
    let minute = parse_u(14, 16)?;
    let second = parse_u(17, 19)?;

    if !(1..=12).contains(&month) { return None; }
    if !(1..=31).contains(&day) { return None; }
    if hour > 23 { return None; }
    if minute > 59 { return None; }
    if second > 60 { return None; }  // tolerate leap-second slot
    Some(DateTimeUtc { year, month, day, hour, minute, second })
}
```

**长度计算**：`"YYYY-MM-DDTHH:MM:SSZ"` = 4+1+2+1+2+1+2+1+2+1+2+1 = 20 字符（不是 19）。注意 plan 头部"19 字符"是错的；实现以 20 为准。

## 风险

| 风险 | 缓解 |
|---|---|
| 解析者期望接受 `+00:00` 当 UTC | 严格 `Z`：plan 在 "非目标" 明确，doc-comment 说明 |
| `s.len()` 与 `bytes.len()` 在含 multi-byte 字符时不一致 | 解析器在第一步 `len() != 20` 拒绝；一旦通过该闸，剩余 19 个固定 ASCII 字符的索引访问安全（`bytes[4] != b'-'` 断言确保非 ASCII 提前 reject） |
| 二月 30 日 / 4 月 31 日通过 day ≤ 31 检查 | 不验证 calendar 规则——交给上层；DXF round-trip 不会产出 illegal date |
