# R46-E2: AC1018 read_dwg top-level wiring + entity recovery

> 起稿：2026-04-29
> 前置：R46-A（encrypted metadata）+ R46-B（LZ77）+ R46-C（page map）
> + R46-D（section descriptors map）+ R46-E1（per-section payload
> reassembly）落地。
> R46-E2 把前面所有砖头串起来：让顶层 `read_dwg(sample_AC1018_bytes)`
> 走完整的 AC1018 → AC1015 桥接 → `build_pending_document` →
> `resolve_document` → `enrich_with_real_entities` 链路，最终返回
> `Ok(CadDocument)`。R46-F 再做 baseline_m3b ratchet。

## 1. 范围

新增 / 修改：

- **error.rs**：`DwgReadError` 新增 `Ac1018Decode { stage, reason }`
  变体，承接 R46-A/C/D/E1 的子 brick 错误。提供 `Display` 与 string
  payload 转换（不要求 source chain，因为 R46 brick 的错误类型互不
  兼容；一律靠 `to_string()` 摘要）。
- **lib.rs**：新增私有辅助 `read_dwg_ac1018(bytes: &[u8]) ->
  Result<CadDocument, DwgReadError>`，内部依次：
  1. `parse_ac1018_encrypted_metadata` → R46-A
  2. `parse_ac1018_page_map` → R46-C
  3. `parse_ac1018_section_map` → R46-D
  4. 对每个 R46-D descriptor 调 `read_ac1018_section_payload` →
     R46-E1
  5. 用 R46-D 的 `descriptor.section_id` 反推 record_number（
     `KnownSection::from_record_number` 的反向查找——与 AC1015 的
     `record_number ∈ {0,1,2,3,4,5}` 对齐），合成
     `(DwgFileHeader, SectionMap, Vec<Vec<u8>>)`
  6. 复用 `build_pending_document` + `resolve_document` +
     `enrich_with_real_entities`
- **lib.rs `read_dwg`**：在 `DwgFileHeader::parse` 调用之前先
  `sniff_version`，按版本分发：
  - `Ac1015` → 维持当前路径（零回归）
  - `Ac1018` → 调 `read_dwg_ac1018`
  - 其他 → `Err(UnsupportedVersion)`（与现状对齐）
- **file_header.rs**：保持 `section_count_offset(Ac1018)` 仍返回
  `UnsupportedHeaderLayout`。R46-E2 **不动** AC1015 路径。

R46-E2 不修改：
- `crates/h7cad-native-facade/src/lib.rs`（已经在 R48 接通到
  `read_dwg`，AC1018 走通后自动获益）
- `src/io/mod.rs`（H7CAD 主 bin 仍 acadrust-only；切默认 backend
  是单独的 long-term 工作）

## 2. AC1018 → AC1015 桥接逻辑

### 2.1 record_number 映射

AC1015 的 `KnownSection::from_record_number` 给出唯一映射：

| record_number | KnownSection name |
|---:|---|
| 0 | `AcDb:Header` |
| 1 | `AcDb:Classes` |
| 2 | `AcDb:Handles` |
| 3 | `AcDb:ObjFreeSpace` |
| 4 | `AcDb:Template` |
| 5 | `AcDb:AuxHeader` |

R46-E2 加反向查找 `KnownSection::record_number_from_name(name) ->
Option<u8>`。AC1018 descriptor 的 `name` 严格匹配上表时返回 `Some`，
否则 `None`（descriptor 仍合法，但不会进入 AC1015 风格的
`SectionDescriptor`，因为那是历史 record_number 槽位）。

### 2.2 SectionMap 合成

每个 AC1015 风格的 `SectionDescriptor` 有：
- `index: u32`：合成时用 0..N 顺序
- `record_number: u8`：来自上述反向查找
- `offset: u32` / `size: u32`：原 AC1015 用 `offset` 指向
  `bytes[offset..offset+size]` 内的 raw 区域。R46-E2 由于已经把
  `read_ac1018_section_payload` 的输出当作单独的 `Vec<u8>` 放在
  `Vec<Vec<u8>>` 中传给 `build_pending_document`，**所以
  `offset/size` 字段在 AC1018 路径上是诊断字段**，可填 `(0, payload.len())`。

`build_pending_document` 用 `SectionMap.descriptors[i].record_number`
配 `KnownSection::from_record_number` 做识别，所以只要 record_number
正确，offset/size 是占位也无伤大雅。

### 2.3 DwgFileHeader 合成

`build_pending_document` 需要 `&DwgFileHeader` 取 `version` /
`section_count`。AC1018 路径合成：
- `version = DwgVersion::Ac1018`
- `section_count = (only-the-mapped-descriptors).len() as u32`
- `magic = "AC1018"`
- `section_directory_offset = 0`（不被 AC1018 路径消费）

### 2.4 SectionMap::read_section_payloads 旁路

`SectionMap::read_section_payloads(bytes)` 会按 `descriptor.offset/
size` 切 `bytes`。AC1018 路径**不**调用它——直接把
`Vec<read_ac1018_section_payload(...)>` 喂给 `build_pending_document`。

## 3. 错误模型

R46-A/C/D/E1 的错误类型互不兼容，且都不实现 `From<X> for
DwgReadError`。R46-E2 加新变体：

```rust
pub enum DwgReadError {
    // ... existing ...
    /// AC1018 sub-brick (R46-A/C/D/E1) decode failure. `stage`
    /// identifies which brick raised the error; `reason` is the
    /// brick's own error rendered to a string.
    Ac1018Decode {
        stage: &'static str,
        reason: String,
    },
}
```

`stage` 取值：
- `"encrypted_metadata"`（R46-A）
- `"page_map"`（R46-C）
- `"section_descriptor_map"`（R46-D）
- `"section_payload"`（R46-E1，附带 section name 也可，后续优化）

不引入 `From` impl —— `read_dwg_ac1018` 用 `.map_err(|e|
DwgReadError::Ac1018Decode { stage: "...", reason: e.to_string() })?`
显式转换，让出错位置在源码里清晰可见。

## 7. 风险

- **`section_id == 0` 的空 section**：R46-D 实测显示 sample 第 0 个
  descriptor 名字是空字符串 `""`，`KnownSection::record_number_from_name("")`
  返回 `None`，会被静默跳过——这正是想要的行为。
- **R46-D 的 13 个 descriptors 比 AC1015 的 6 个 record_number 多**：
  非映射的 descriptors（AppInfo / Preview / SummaryInfo / RevHistory
  / AppInfoHistory 等）会被静默 drop，不进入 SectionMap。这与
  R46-E2 的目标一致：先让 AC1015 风格的 6 个核心 section 跑通，其它
  作为 R46-F 之后的扩展点。
- **R46-E1 的 cap 已放宽到 16 MiB**：单 section payload 最大可达
  16 MiB × 13 = 208 MiB（理论），实测 sample 上 AcDb:Header 约
  29 KiB，所有 13 个 descriptor 加起来远低于 1 MiB。R46-E2 不引入
  额外 allocator pressure。
- **build_pending_document 假设 `section_records.len() ==
  payloads.len()`**：R46-E2 必须保证 `SectionMap.descriptors` 与
  `payloads` 严格 1:1 对齐，否则 zip 会丢数据。
- **enrich_with_real_entities 调用 `pending.handle_offsets`**：
  R46-E2 喂给 `build_pending_document` 的 `AcDb:Handles` payload
  必须**与 AC1015 上同名 section 的 payload 在二进制层兼容**。
  ACadSharp 的 `parseHandleMap` 在 AC1015 / AC1018 上格式相同
  （modular char + 7-bit chunks），所以 R46-E1 解出的 raw bytes
  应该可以直接喂给 `parse_handle_map`。如果实测发现 handle_offsets
  为空，先打开 R46-E2 的 sample test 诊断输出确认 payload 是否合理。
- **resolve_document / enrich_with_real_entities 假设 AC1015**：
  这两个函数当前没有版本分支，对 AC1018 应该也工作（pending 数据
  结构是 version-agnostic 的）。R46-E2 不强制等 entity recovery 数量
  非零（保留 `> 0` 的 weak gate）。

## 9. R46-F 衔接

R46-F 把 R46-E2 完成后 `sample_AC1018.dwg` 上的真实 entity recovery
数量作为 baseline，加入 `real_dwg_samples_baseline_m3b` 的 ratchet
列表。当前 baseline_m3b 显式对 AC1018 走 soft-skip / "structural
error tolerated"；R46-F 会把这个跳过逻辑去掉，让 AC1018 在 baseline
上与 AC1015 同等待遇。

R46-F 估时 0.5h，主要是测试维护，不涉及 parser 修改。
